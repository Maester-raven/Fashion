# 3.1.1 Optimization History

This document summarizes the main technical route explored before the current 3.1.1 deployable RTMDet-Ins-L A/epoch5 version.

## 1. Early route

The early exploration included:

- Mask2Former proposal
- SAM-HQ refine / automatic proposal
- DINOv2 reranker
- Coarse/fine verifier

Main issues observed in this route:

- High proposal redundancy
- Low precision
- Obvious confusion among dress / top / skirt / pants
- Complex engineering chain for stable deployment

## 2. Mask R-CNN baseline

Mask R-CNN was used as an early instance segmentation baseline, especially on DeepFashion2.

Observations:

- It achieved relatively high recall on DeepFashion2.
- Precision remained weak.
- Misclassification was still visible for accessory, skirt, and dress-like categories.

## 3. YOLOv8-seg comparison

YOLOv8-seg was evaluated as a lightweight instance segmentation comparison route.

Conclusion:

- It was useful as a lightweight reference.
- Overall quality did not match the RTMDet-Ins mainline for this 8-class fashion parsing task.

## 4. RTMDet-Ins-L mainline

The final 3.1.1 route uses RTMDet-Ins-L as the main model family. The main steps were:

- Introduce Fashionpedia
- Unify the task into 8-class fashion instance segmentation
- Perform hard-image fine-tuning
- Search class-specific thresholds
- Add mask overlap conflict-resolution rules
- Reach the stage-one acceptance target

## 5. TensorRT deployment optimization

The deployment path was optimized for clean-server reproducibility:

- A-only ONNX export
- TensorRT FP16 engine built locally on the target machine
- Removed hard dependency on the D1 checkpoint
- Fixed inference compatibility when the A-only engine has no `d1_cls_*` outputs
- Clean-server single-image demo validation passed
